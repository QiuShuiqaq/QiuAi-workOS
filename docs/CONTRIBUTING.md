# QiuAI WorkOS 开发规范

## 基本原则

QiuAI WorkOS 是企业数字劳动力平台。任何产品、设计、架构和代码决策都应优先服务于以下原则：

- 岗位优先：用户管理的是 AI 岗位，不是 Agent。
- 业务优先：企业描述业务目标，系统承担技术复杂度。
- 模板优先：每个客户项目都要尽量沉淀为可复用行业模板。
- 企业级优先：权限、审批、审计、成本和监控是基础能力。
- 可追踪优先：AI 岗位的每次执行都要能解释、回放和归因。

## 文档约定

核心文档位于 `docs/`：

- `VISION.md`：产品愿景和长期定位
- `ARCHITECTURE.md`：系统架构和模块边界
- `DOMAIN_MODEL.md`：领域模型和核心实体
- `ROADMAP.md`：版本规划和阶段目标
- `BUSINESS_MODEL.md`：商业模式和增长路径
- `ENTITLEMENT_MODEL.md`：套餐、订阅、权益和配额模型
- `PROJECT_STRUCTURE.md`：Monorepo、应用、共享包和服务端模块结构
- `FRONTEND_UI_ARCHITECTURE.md`：前端技术栈、Ant Design 生态和 QiuAI UI 组件规范
- `LOCAL_DEVELOPMENT.md`：本地开发工具、缓存和环境约束
- `CONTRIBUTING.md`：开发规范

当修改以下内容时，必须同步更新相关文档：

- 产品定位
- 核心领域模型
- 关键架构边界
- 版本目标
- 商业化策略
- 套餐权益和收费边界
- 企业级治理能力

重大技术决策应新增 ADR，建议路径为 `docs/decisions/ADR-XXX-title.md`。

## 领域语言

对外产品语言优先使用：

- 企业
- 部门
- 岗位
- 任务
- 结果
- 绩效
- 成本
- 审批
- 审计

底层技术语言只在高级模式、开发者文档或实现层出现：

- Agent
- Workflow
- Prompt
- Tool
- MCP
- Memory
- Model

普通企业用户界面不应把 Agent 和 Workflow 作为主要操作对象。

## 产品设计规范

新增功能前先回答：

- 这个功能服务于哪个 AI 岗位？
- 这个功能帮助企业完成什么业务目标？
- 企业用户是否需要理解底层 AI 技术？
- 这个能力能否沉淀为模板？
- 是否需要权限、审批、审计或成本记录？

如果一个功能不能回答以上问题，应重新评估优先级。

## 套餐与权益规范

个人版和企业版必须共用同一套平台内核，通过 Plan、Subscription、Entitlement 和 UsageMeter 控制功能和额度。

要求：

- 个人版是免费 Personal Workspace，不支持 Department、成员邀请、企业级权限、审批和审计。
- 企业版是收费 Enterprise Workspace，以 Organization 为核心，并支持 Department、成员、权限、审批、审计和成本治理。
- Web、PC 和移动端只能展示权益状态，不能作为最终校验来源。
- 后端所有受限写操作都必须检查 Entitlement 和 UsageMeter。
- 权益不足或配额超限时，API 必须返回统一结构化错误。
- 不为个人版和企业版维护两套业务模型。

## 架构规范

### 多租户

所有企业数据必须有明确租户边界。新增数据模型时必须考虑：

- tenantId 或 organizationId
- 数据访问权限
- 审计记录
- 删除和归档策略

### 岗位中心

任务、执行、成本、日志和绩效应尽量归因到 RoleInstance。

不要让 Task、WorkflowRun 或 AgentRun 脱离岗位上下文成为主要业务对象。

### 底层能力可替换

模型供应商、Agent Framework、Workflow Engine、向量数据库和工具协议都应可替换。

上层业务对象不应直接依赖某个模型或框架的专有概念。

### 企业治理默认存在

涉及业务结果输出、外部系统写入、客户触达、内容发布或高成本操作时，必须考虑：

- 权限控制
- 人工审批
- 审计日志
- 成本预算
- 失败回滚或补偿

## 代码规范

当前仓库尚未固定技术栈。确定技术栈后，应补充对应规范。

通用要求：

- 代码命名使用清晰业务语义。
- 公共接口要有类型定义和输入校验。
- 复杂业务逻辑要有测试覆盖。
- 不提交密钥、Token、客户数据或本地环境文件。
- 不提交一次性临时文件、构建产物或调试输出。
- 不用注释复述代码，只记录非显而易见的业务原因或技术约束。

## 测试规范

新增能力至少考虑以下测试层级：

- 领域模型测试：状态流转、权限边界、成本归因
- 服务层测试：岗位安装、任务创建、审批、执行回调
- 集成测试：模型、工具、知识库和任务执行链路
- 端到端测试：企业用户从安装岗位到查看结果的主路径

AI 相关能力还需要记录：

- 输入样例
- 期望输出
- 失败样例
- 人工验收标准
- 质量指标

## Prompt、Workflow 和 Agent 规范

底层 AI 能力必须可版本化、可回放、可评估。

要求：

- Prompt 有版本号和适用场景。
- Workflow 有输入输出契约。
- Agent 有明确职责边界。
- Tool 调用有权限边界。
- 高风险步骤必须支持人工审批。
- 每次执行记录模型、参数、成本和输出摘要。

## 提交流程

建议流程：

1. 明确业务目标和影响范围。
2. 更新或确认相关文档。
3. 实现最小可验证改动。
4. 补充测试或验收样例。
5. 检查权限、审计、成本和日志影响。
6. 提交代码评审。

## PR 检查清单

提交前确认：

- 是否仍然符合“岗位优先”的产品抽象？
- 是否避免把普通企业用户暴露给底层 Agent/Workflow 复杂度？
- 是否更新了相关文档？
- 是否有必要新增 ADR？
- 是否有测试或明确的人工验收方式？
- 是否处理了权限、审批、审计、成本和失败状态？
- 是否避免提交密钥、客户数据和临时文件？
